Estrutura obrigatória de migration:
1. Header de comentário com nome, data, "Idempotente: pode ser executado múltiplas vezes sem erro."
2. CREATE TABLE IF NOT EXISTS com apenas PRIMARY KEY
3. ALTER TABLE ADD COLUMN IF NOT EXISTS para cada coluna (NOT NULL com DEFAULT)
4. DROP CONSTRAINT IF EXISTS + ADD CONSTRAINT para cada constraint (idempotência)
5. CREATE INDEX IF NOT EXISTS para cada índice
6. ALTER TABLE ENABLE ROW LEVEL SECURITY
7. DROP POLICY IF EXISTS + CREATE POLICY para cada política
8. NOTIFY pgrst, 'reload schema'; no final

Funções RLS disponíveis no banco:
- get_my_role() → TEXT   (retorna 'superuser' | 'dev' | 'adm' | 'auditor' | NULL)
- get_my_company_id() → UUID
- auth.uid() → UUID

Padrão Pattern 1 — superuser/dev full access (toda tabela nova precisa):
```sql
DROP POLICY IF EXISTS "superuser_dev_{table}_full" ON {table};
CREATE POLICY "superuser_dev_{table}_full" ON {table}
  USING (get_my_role() IN ('superuser', 'dev'))
  WITH CHECK (get_my_role() IN ('superuser', 'dev'));
```

Padrão Pattern 2 — company + creator (para checklist_executions):
- SELECT: adm vê todas da empresa; auditor vê as próprias
- INSERT: WITH CHECK (created_by = auth.uid())
- UPDATE: USING + WITH CHECK (created_by = auth.uid())

Padrão Pattern 3 — subquery via FK (para checklist_answers derivar de checklist_executions):
```sql
EXISTS (
  SELECT 1 FROM checklist_executions e
  WHERE e.id = checklist_answers.execution_id
    AND e.created_by = auth.uid()
)
```
